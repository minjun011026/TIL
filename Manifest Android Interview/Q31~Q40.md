# Q31 애플리케이션 용량을 어떻게 줄이나요?

| **사용하지 않는 리소스 제거하기**   
사용되지 않는 리소스는 APK/AAB 용량을 증가시킨다.   
Android Studio의 Lint와 같은 도구를 이용해 식별할 수 있다.   
사용하지 않는 리소스를 제거한 후, build.gradle에서 `shrinkResources`를 활성화하여 빌드 프로세스 중 사용되지 않는 리소스를 자동으로 제거하도록 한다.

```kotlin
android {
    buildTypes {
        release {
            minifyEnabled true
            shrinkResources true
        }
    }
}
```

| **R8로 코드 축소 활성화하기**   
R8은 사용되지 않는 클래스와 메서드를 제거한다.

```kotlin
buildTypes {
    release {
        isMinifyEnabled = true
        proguardFiles(
            getDefaultProguardFile("proguard-android-optimize.txt"),
            "proguard-rules.pro"
        )
    }
}
```

| **리소스 최적화 사용하기**   
이미지 및 XML 파일과 같은 리소스를 최적화하여 앱 용량을 줄일 수 있다.

+ Vector Drawable : 확장 가능한 그래픽을 위해 래스터 이미지(PNG, JPEG 등) 대신 용량을 덜 차지하는 Vector Drawable 사용
+ 이미지 압축 : TinyPNG/ImageMagick 과 같은 도구를 사용하여 눈에 띄는 품질 손상없이 래스터 이미지를 압축한다.
+ WebP 형식 : 이미지를 PNG 또는 JPEG보다 압축률이 높은 WebP 형식으로 변환한다.

```kotlin
android {
    defaultConfig {
        vectorDrawables.useSupportLibrary = true
    }
}
```

| **Android App Bundles(AAB) 사용하기**   
AAB 형식을 전환하면 Google Play가 개별 기기에 맞는 최적화된 APK를 제공할 수 있다.

```kotlin
android {
    bundle {
        density {
            enableSplit = true
        }
        abi {
            enableSplit = true
        }
        language {
            enableSplit = true
        }
    }
}
```

위와 같은 방식으로 화면 밀도, CPU 아키텍처, 언어 등 특정 구석에 필요한 리소스와 코드만 포함하여 앱 용량을 줄인다.

| **불필요한 의존성 제거하기**   
Android Studio의 Gradle Dependency Analyzer를 사용하여 무겅누 라이브러리 및 전이 의존성을 식별할 수 있다.

| **네이티브 라이브러리 최적화하기**   
앱에 네이티브 라이브러리가 포함된 경우 다음 전략을 사용하여 앱 용량을 줄일 수 있다.

+ 사용하지 않는 아키텍처 제외 : build.gradle의 `abiFilters` 옵션을 사용하여 필요한 ABI만 포함한다.
+ 디버그 심볼 제거 : stripDebugSymbols를 사용하여 네이티브 라이브러리에서 디버깅 심볼을 제거한다.

| **동적 기능(Dynamic Features) 사용하기**   
동적 기능 모듈을 사용하면 자주 사용하지 않는 기능을 주문형 모듈(해당 기능이 필요한 경우 설치)로 분리하여 앱을 모듈화할 수 있다.   
초기 다운로드 용량을 줄이는데 도움

| **앱 내 대용량 에셋 피하기**

+ 비디오나 고해상도 이미지와 같은 대용량 에셋은 콘텐츠 전송 네트워크(CDN)에 호스팅하고 런타임에 동적으로 로드
+ 미디어 콘텐츠는 앱과 함께 번들링하는 대신 스트리밍 사용

# Q32 안드로이드 애플리케이션의 프로세스(process)란 무엇이며, 안드로이드 운영 체제는 이를 어떻게 관리하나요?

안드로이드 프로세스는 애플리케이션이 실행되는 환경으로 각 안드로이드 앱은 다른 앱과 격리된 자체 프로세스에서 단일 실행 스레드로 작동하여 시스템 보안, 메모리 관리 및 내결합성을 보장한다.   
안드로이드 프로세스는 리눅스 커널을 사용하여 운영 체제에 의해 관리되며 엄격한 생명주기 규칙을 따른다.   
기본적으로 동일한 어플리케이션의 모든 컴포넌트는 메인 스레드라고 하는 동일한 프로세스 및 스레드에서 실행된다.

| **안드로이드에서 프로세스 작동 방식**

+ 안드로이드 애플리케이션 실행시 운영 체제는 리눅스 `fork()` 시스템 함수를 호출하여 해당 앱을 위한 새 프로세스 생성
+ 각 프로세스는 Dalvik 또는 ART 가상 머신의 고유 인스턴스에서 실행되어 안전하고 독립적인 실행 보장
+ 안드로이드는 각 프로세스에 고유한 리눅스 사용자 ID(UID)를 할당하여 권한 제어 및 파일 시스템 격리를 포함한 엄격한 보안 경계 적용

| **애플리케이션 컴포넌트와 프로세스 연결**   
기본적으로 안드로이드 애플리케이션의 모든 컴포넌트는 동일한 프로세스 내에서 실행되며 대부분의 애플리케이션은 이 표준을 따른다.      
개발자는 AndroidManifest.xml 파일의 `android:process` 속성을 사용하여 프로세스 할당을 스텀할 수 있다.   
activity, service, receiver, provider같은 컴포넌트에 적용될 수 있어 컴포넌트가 별도의 프로세스에서 실행되거나 선택적으로 프로세스를 공유할 수 있도록 한다.
application에서 해당 속성 적용시 모든 컴포넌트의 기본 프로세스 정의 가능

```xml

<service
        android:name=".MyService"
        android:process=":remote"/>
```

:remote라는 별도의 프로세스에서 실행되어 독립적인 작동과 향상된 내결합성 가능케 함

+ 다른 애플리케이션의 컴포넌트가 동일한 리눅스 사용자 ID를 가지고 동일한 인증서로 서명된 경우 동일한 프로세스 공유 가능
+ 안드로이드는 시스템 리소스 요구에 따라 프로세스를 동적으로 관리, 필요시 우선순위가 낮은 프로세스 종료
+ 안드로이드 시스템은 연관된 컴포넌트가 작업을 수행해야 할 때 프로세스를 다시 시작하여 최적의 시스템 성능과 사용자 경험 보장

| **프로세스와 앱 생명주기**   
안드로이드에서 프로세스는 다은과 같은 우선순위를 가진다.

1. Foreground Process : 사용자와 활발하게 상호 작용하며 실행 중인 프로세스, 가장 높은 우선순위의 프로세스로 거의 종료 안됨
2. Visible Process : 사용자에게 보이지만 활발하게 상호 작용하지 않는 프로세스(ex.다이얼로그 뒤의 Activity)
3. Service Process : 데이터 동기화/음악재생같은 작업을 수행하는 백그라운드 Service를 실행하는 프로세스
4. Cached Process : 더 빠른 재실행을 위해 메모리에 유지되는 유휴 프로세스, 가장 우선순위가 낮고 메모리가 부족할 때 가장 먼저 종료

| **보안 및 권한**

+ 각 안드로이드 프로세스는 리눅스 보안 모델을 사용하여 샌드박스 처리되어 엄격한 권한 기반 접근 제어 시행
+ 안드로이드 권한 시스템을 통해 명시적으로 권한이 부여되지 않는 한 다른 프로세스의 데이터에 접근할 수 없도록 보장
+ 안드로이드의 멀티태스킹 환경의 기본, 시스템 안정성과 데이터 개인 정보 보호 모두 지원

## 실전질문

Q) 서로 다른 안드로이드 컴포넌트들을 별도의 프로세스에서 실행해야 하는 애플리케이션을 개발 중이라고 해봅시다. AndroidManifest에서 이를 어떻게 구성하며, 여러 프로세스를 사용할 때의 잠재적인 단점은
무엇인가요?   
A) AndroidManifest.xml의 각 컴포넌트의 android:process 속성에서 프로세스를 설정한다. 서로 다른 프로세스에 있는 컴포넌트들이 데이터를 주고받기 위해 IPC 메커니즘이 사용되어 성능
오버헤드가 발생한다. 또한 각 프로세스가 생성되어 메모리 사용량이 증가하고 싱글톤/전역 객체는 각 프로세스마다 존재하게 되어 값의 일관성을 보장하는데 별도의 로직이 필요해진다.

Q) 안드로이드는 메모리가 부족할 때 어떤 프로세스를 종료할지 결정하기 위해 우선순위 기반 프로세스 관리 시스템을 사용합니다. 시스템이 프로세스 우선순위를 어떻게 정하는지, 그리고 중요한 프로세스가 종료되는 것을
방지하기 위해 개발자가 따라야 할 전략은 무엇인지 설명해 주세요.   
A) oom_adj_score라는 메모리 부족 점수를 기반으로 사용중인 프로세스의 우선순위를 결정한다. 중요한 프로세스가 종료되는 것을 막기위해 지속적인 Notification이나
`startForegroundService()`를 통해 서비스를 실행하여 포그라운드 서비스로 실행한다.

### Activities, Services, Broadcast Receivers, Content Providers가 안드로이드의 4대 주요 컴포넌트라고 불리는 이유는 무엇인가요?

각 컴포넌트들이 4대 컴포넌트로 불리는 것은 프로세스와 밀접한 관려이 있기 때문이다.

1. **Activity**
    + Activity는 사용자 인터페이스가 있는 단일 화면을 나타낸다.
    + 사용자 상호 작용의 진입점이며 안드로이드 프로세스 생명주기와 밀접하게 연결되어 있다.
    + 사용자가 앱을 열면 시스템은 앱의 프로세스에서 Activity를 시작한다.
    + 프로세스가 종료되면 Activity는 소멸되고 앱을 다시 시작하면 새 프로세스가 생성된다.
2. **Service**
    + Service는 사용자 인터페이스 없이 백그라운드 작업 수행
    + Service는 앱의 Manifest에 지정된 `android:process` 속성에 따라 앱과 동일한 프로세스 또는 별도의 프로세스에서 실행 가능
3. **BroadCast Receiver**
    + 애플리케이션이 시스템 전체 브로드캐스트 메시지를 수신하고 응답 가능
    + 앱이 실행중이 아니더라도 트리거되어 필요한 경우 안드로이드 시스템이 해당 프로세스를 시작하도록 한다.
4. **Content Provider**
    + 공유 애플리케이션 데이터를 관리하여 앱이 중앙 집중식 데이터베이스에서 읽거나 쓸 수 있도록 한다.
    + IPC를 허용하므로 다른 애플리케이션 간에 데이터를 공유하는데 사용될 수 있으며, 안드로이드 시스템이 프로세스를 안전하고 효율적으로 관리하도록 요구

이 4대 컴포넌트는 각각이 안드로이드 OS에서 자체 전용 프로세스를 가질 수 있음을 의미한다.    
전용 프로세스에서 실행되도록 구성될 수 있으므로 시스템 수준 기능을 얻게 되어 다른 안드로이드 컴포넌트에 비해 더 강력하고 독립적이다.

참고   
내결합성 : 시스템의 일부 구성 요소에 결함(fault)이나 오류(error)가 발생하더라도, 시스템 전체가 중단되지 않고 정상적인 기능을 계속해서 수행할 수 있는 능력

# Q33 View 생명주기를 설명해주세요

View Life Cycle은 View가 생성되고 Activity/Fragment에 연결되고, 화면에 표시되고 최종적으로 소멸되거나 분리되는 동안 거치는 생명주기 이벤트를 나타낸다.

1. View 생성(onAttachedToWindow)
    + View가 하드코딩 방식으로 인스턴스화되거나 XML 레이아웃에서 인플레이션되는 단계
    + 리스너 설정 및 데이터 바인딩과 같은 초기 설정 가업이 수행된다.
    + `onAttachedToWindow()` 메서드는 View가 부모 뷰에 추가되고 화면 렌더링을 할 준비를 마쳤을 때 트리거된다.
2. Layout 단계(onMeasure, onLayou)
    + View의 크기와 위치를 측정한다.
    + `onMeasure()` 메서드는 레이아웃 매개변수와 부모 제약 조건에 따라 View의 너비와 높이를 결정한다.
    + 측정된 후 `Layout()` 메서드는 View를 부모 내에 배치하여 화면에 표시될 위치를 최종 결졍한다.
3. Drawing 단계(onDraw)
    + 크기와 위치가 최종 결정된 후 `onDraw()` 메서드는 텍스트나 이미지와 같은 View의 내용을 Canvas에 렌더링한다.
    + 커스텀 View는 해당 메서드를 재정의하여 커스텀 드로잉 로직 구현 가능
4. Event 처리(onTouchEvent, onClick)
    + 상호작용하는 View는 이 단계에서 터치 이벤트, 클릭, 제스처와 같은 사용자 상호 작용을 처리한다.
    + `onTouchEvent()` 및 `onClick()`과 같은 메서드는 이러한 이벤트를 처리하고 사용자 입력에 대한 View의 응답을 정의하는데 사용된다.
5. View 분리(onDetachedFromWindow)
    + View가 화면과 부모 ViewGroup에서 제거될 때(ex.Activity/Fragment의 소멸), `onDetachFromWindow()` 메서드가 호출된다.
    + 리소스를 정리하거나 리스너를 분리하는데 이상적
6. View 소멸
    + View가 더이상 사용되지 않으면 가비지 컬렉션된다.
    + 개발자는 메모리 누수를 방지하고 성능을 최적화하기 위해 이벤트 리스너나 백그라운드 작업과 같은 모든 리소스가 적절하게 해재되었는지 확인해야 한다.

### View의 findViewTreeLifecycleOwner() 함수는 어떤 역할을 하나요?

`findViewTreeLifecycleOwner()`는 View 클래스의 확장 함수로 View 트리 계층을 거슬러 올라가 View 트리에 연결된 가장 가까운 LifecycleOwner를 찾아 반환한다.   
여기서 말하는 LifecycleOwner는 일반적으로 Activity, Fragment 또는 LifecycleOwner를 구현하는 커스텀 컴포넌트와 같은 호스팅 컴포넌트의 생명주기 범위를 나타낸다.   
만약 못찾으면 null 반환

**사용하는 이유**   
LiveData, ViewModel 또는 LifecycleObserver와 같은 생명주기 인식 요소와 상호작용해야 하는 커스텀 View나 서드파티 컴포넌트로 작업할 때 유용하다.   
이를 통해 View는 호스팅 Activity나 Fragment에 대한 명시적인 의존성 없이 연관된 생명주기에 접근할수 있다.

`findViewTreeLifecycleOwner()`를 사용하면 다음을 보장할 수 있다.

+ 생명주기를 인식하는 컴포넌트(LiveData 등)가 올바른 생명주기에 제대로 바인딩된다.
+ 생명주기가 끝나면 관찰자가 정리되도록 하여 메모리 누수를 방지한다.

**사용 사례**

1. Custom View : 커스텀뷰 내의 생명주기 인식(lifecycle-aware, 생명주기와 연결되어 생명주기의 변화에 따라 동작이 결정되는) 컴포넌트가 LifecycleObserver, LiveData와
   같은 생명주기 관찰자를 관찰하거나 리소스를 관리할 수 있도록 한다.
2. 서드파티 라이브러리 : 재사용 가능한 UI 컴포넌트가 명시적인 생명주기 관리없이 생명주기 인식 리소스와 상호작용할 수 있도록 한다.
3. 로직 결합도 분리(Decoupling Logic) : View가 View 트리에서 자체 LifecycleOwner를 독립적으로 획득하도록 하여 부모 클래스와의 결합도를 줄이는데 도움이 된다.

참고   
호스팅 컴포넌트 : 일반적으로 더 작은 구성 요소(예: View, Fragment, ViewModel 등)를 담고, 이들의 생명 주기와 컨텍스트를 제공하고 관리하는 상위(Upper-level) Android
프레임워크 컴포넌트(ex. Activity, Fragment)

# Q34 View와 ViewGroup의 차이점은 무엇인가요?

| **View란?**   
View는 화면에 표시되는 직사각형 형태의 UI 구현하는 최소 단위의 단일 컴포넌트이다.   
Button, TextView와 같은 모든 UI 컴포넌트의 기본 클래스가 되고 각 View는 화면에 렌더링되면 터치/키 이벤트와 같이 애플리케이션 사용자와의 상호작용을 처리한다.

| **ViewGroup이란?**   
ViewGroup은 여러 View 또는 다른 ViewGroup 요소를 포함하는 일종의 컨테이너이다.   
이는 LinearLayout, RelativeLayout, ConstraintLayout과 같은 레이아웃의 기본이 되는 클래스이다.   
ViewGroup은 자식 뷰의 레이아웃과 위치를 관리하며 화면에서 자식 View들의 사이즈가 측정되고 그려지는 방식을 정의한다.

| **주요 차이점**

1. 목적
    + View는 콘텐츠를 표시하거나 사용자와 상호작용하도록 설계된 단일 UI 요소
    + ViewGroup은 여러 자식 뷰를 구성하고 관리하기 위한 컨테이너
2. 계층
    + View는 UI 계층 구조의 리프 노드이다. 따라서 다른 뷰를 포함할 수 없다.
    + ViewGroup은 여러 자식 뷰 또는 다른 ViewGroup 요소를 포함할 수 있는 브랜치 노드이다.
3. 레이아웃 추적
    + View는 레이아웃 매개변수에 의해 정의된 자체적인 크기와 위치를 갖는다.
    + ViewGroup은 LinearLayout 또는 ConstraintLayout과 같이 정의된 레이아웃 규칙을 사용하여 자식 뷰의 크기와 위치를 결정한다.
4. 상호작용 핸들링
    + View는 터치 및 키 이벤트를 처리할 수 있다.
    + ViewGroup은 onInterceptTouchEvent와 같은 메서드를 사용하여 자식의 이벤트를 가로채고 관리할 수 있다.
5. 성능
    + ViewGroup은 계층 구조로 인해 렌더링에 복잡성을 더한다.
    + 중첩된 ViewGroup을 과도하게 사용하면 렌더링 시간 증가 및 UI 업데이트 지연과 같은 성능 문제가 발생할 수 있다.

## 실전 질문

Q) View 생명주기에서 requestLayout(), invalidate(), postInvalidate()가 어떻게 작동하는지 설명하고 각각 언제 사용해야 하나요?   
A)   
requestLayout()

+ Layout 단계(onMeasure/onLayout)을 다시 하도록 요청하는 함수로 View의 크기나 위치가 변경되었을 때, View 계층 구조의 최상단부터 다시 측정하고 배치하도록 요청하여, View의 최종
  크기와 위치를 다시 계산한다.
+ View의 크기나 위치가 변하는 작업을 수행할 때 호출, 비용 많이 드는 작업이므로 많이 호출하면 성능 떨어짐

invalidate()

+ Drawing 단계만 다시 실행하도록 요청하는 함수로 View의 외관(텍스트나 색 등)만 변경되고 위치나 크기는 변경되지 않았을 때 시스템에게 다시 그리도록 요청한다.
+ View의 내용만 바뀌었을 때 사용(텍스트/색 변경 등), 반드시 메인스레드에서 호출되어야함

postInvalidate()

+ invalidate()와 동일하게 Drawing 단계를 다시 요청하지만 백그라운드 스레드에서 안전하게 호출할 수 있도록 해준다. 내부적으로 Handler를 사용하여 요청을 메인 스레드의 메시지 큐에 넣어
  invalidate() 호출을 지연시킨다.
+ 백그라운드 스레드에서 비동기 작업이 완료된 후 View의 UI를 업데이트해야할 때 사용, 백그라운드 스레드에서 이미지 비트맵을 생성한 후 View에 적용할 때 사용한다.

Q) View 생명주기는 Activity 생명주기와 어떻게 다르며, 효율적인 UI 렌더링을 위해 둘 다 이해하는 것이 왜 중요한가요?   
A) 관리 대상과 범위가 다르다. View는 화면 구성 요소(UI element)를 Activity는 앱의 화면과 시스템 자원을 관리한다. 또한 View는 화면에 붙고(onAttachedToWindow) 그려지고(
onDraw) 떼어지고(onDetachedFromWindow) 사라지는 과정에 초점을 두고 Activity는 화면의 생성(onCreate), 전환(onPause), 종료(onDestroy) 과정에 초점을 둔다.   
이 두 생명주기를 이해하는 것은 메모리 누수 방지와 렌더링 효율 증대 측면에서 중요하다.   
콜백이나 애니메이션이 onDetachedFromWindow에서 정리되지 않으면 Activity가 Destroy되어도 View객체가 메모리에 남아 메모리 누수가 일어날 수 있다.  
또한 Activity가 화면에서 가려질 때, 불필요한 높은 프레임의 애니메이션이나 센서 리스너 등의 View 관련 리소스를 여기서 중지해야 CPU와 배터리 소모를 줄일 수 있고   
onMeasure/onLayout는 비용이 많이 드는 작업으로 View의 생명주기에 대한 이해가 있으면 불필요한 requestLayout 호출 등을 방지할 수 있다.

# Q35. ViewStub이란 무엇이고, 이를 사용하여 UI 성능을 최적화해 본 경험이 있나요?

ViewStub은 명시적으로 필요할 때까지 레이아웃의 인플레이션을 지연시키는데 사용되는 가볍고 보이지 않는 플레이스홀더 뷰이다.   
앱 생명주기 동안 지금 당장 필요하지 않은 뷰를 필요한 시기에 적절하게 인플레이션하여 보어헤드를 피함으로써 성능을 개선하는데 사용된다.

| **ViewStub의 주요 특징**

1. 가벼움 : ViewStub은 인플레이션될 때까지 레이아웃 공간을 차지하거나 리소스를 소비하지 않아 메모리 공간이 최소화된 매우 가벼운 뷰이다.
2. 인플레이션 지연 : ViewStub에 지정된 실제 레이아웃은 `inflate()` 메서드가 호출되거나 ViewStub이 보이게될 때만 인플레이션된다.
3. 일회성 : 한번 인플레이션되면 ViewStub은 뷰 계층 구조에서 인플레이션된 레이아웃으로 대체되며 재사용할 수 없다.

| **ViewStub의 일반적인 사용 사례**

1. 조건부 레이아웃 : ViewStub은 오류 메시지, 진행률 표시줄 또는 선택적 UI 요소와 같이 조건부로 표시되는 레이아웃에 이상적이다.
2. 초기 렌더링 시간 절감 : 복잡하거나 리소스 집약적인 뷰의 인플레이션을 지연시킴으로써 ViewStub은 Activity 또는 Fragment의 초기 렌더링 시간을 개선하는데 도움이 된다.
3. 동적 UI : 필요할 때만 화면에 동적으로 콘텐츠를 렌더링하는데 사용될 수 있어 메모리 사용량을 최적화한다.

| **ViewStub 사용 방법**

```xml

<LinearLayout
        xmlns:android="http://schemas.android.com/apk/res/android"
        android:layout_width="match_parent"
        android:layout_height="match_parent"
        android:orientation="vertical">

    <TextView
            android:id="@+id/title"
            android:layout_width="wrap_content"
            android:layout_height="wrap_content"
            android:text="Main Content"/>

    <ViewStub
            android:id="@+id/viewStub"
            android:layout_width="match_parent"
            android:layout_height="wrap_content"
            android:layout="@layout/optional_content"/>
</LinearLayout>
```

| **ViewStub의 장점**

1. 최적화된 성능 : 늦게 초기화해도 되는 뷰 생성을 지연시켜 메모리 사용량을 줄이고 초기 렌더링 성능을 개선한다.
2. 쉬운 레이아웃 관리 : 뷰를 수동으로 추가하거나 제거하지 않고도 UI 요소를 선택적으로 렌더링함으로써 쉽게 관리 가능
3. 쉬운 사용성 : API 사용이 간단하고 XML 통합이 쉬워 개발자가 쉽게 활용 가능

| **ViewStub의 한계**

1. 일회성 : 일단 인플레이션되면 ViewStub은 뷰 계층 구조에서 제거되며 재사용할 수 없음
2. 제한된 컨트롤 : 플레이스홀더이므로 인플레이션될 떄까지 사용자 상호 작용을 처리하거나 복잡한 작업을 수행할 수 없음

# Q36 커스텀 뷰(Custom Views)는 어떻게 구현하나요?

| **1. 커스텀 View 클래스 생성하기**   
기본 뷰 클래스(View, ImageView, TextView 등)를 확장하는 새 클래스를 정의한다.   
그후 구현하려는 커스텀 동작에 따라 `onDraw()`, `onMeasure()` 과 같은 필요한 생성자 및 메서드를 재정의한다.

| **2. XML 레이아웃에서 커스텀 View 사용하기**   
커스텀 뷰 클래스를 생성한 후 XML 레이아웃 파일에서 직접 참조할 수 있다.   
유의해야하는 점은 커스텀 클래스의 전체 패키지 이름을 정확하게 사용해야 한다.

| **3. 커스텀 속성 추가하기 (선택 사항)**    
res/values 폴더에 새 attr.xml 파일을 만들어 커스텀 속성을 정의할 수 있다.   
이를 통해 XMl 레이아웃에서 뷰의 속성을 커스텀할 수 있다.

```xml
<?xml version="1.0" encoding="utf‑8"?>
<resources>
    <declare‑styleable name="CustomCircleView">
        <attr name="circleColor" format="color"/>
        <attr name="circleRadius" format="dimension"/>
    </declare‑styleable>
</resource>
```

이후 커스텀 뷰 클래스에서 `context.obtainStyledAttributes()`를 사용하여 커스텀 속성 값을 가져올 수 있다.

| **4. 레이아웃 측정 처리하기 (선택 사항)**   
커스텀 뷰가 크기를 측정하는 방식을 수동으로 처리할 수 있다.

## 실전 질문

Q) XML 레이아웃에서 이전 버전과의 호환성을 보장하면서 커스텀 뷰에 커스텀 속성을 효율적으로 적용하려면 어떻게 해야 하나요?    
A) attrs.xml에 커스텀 속성과 기본값을 정의한다. `context.obtainStyledAttributes()`는 다음과 같이 되어있는데

```java
@NonNull
@RavenwoodKeep
public final TypedArray obtainStyledAttributes (@Nullable AttributeSet set,
@NonNull @StyleableRes int[] attrs, @AttrRes int defStyleAttr,
@StyleRes int defStyleRes) {
    return getTheme().obtainStyledAttributes(
        set, attrs, defStyleAttr, defStyleRes
    );
}
```
이것을 통해
```kotlin
someColor = a.getColor(R.styleable.MyView_someColor, defaultColor)
```
+ set : XML 레이아웃에서 뷰에 정의된 속성들의 집합
+ attrs : XML에서 지정한 스타일
+ defStyleAttr : 테마(Theme)에서 defStyleAttr에 매핑된 스타일
+ defStyleRes : 코드에서 넘긴 defStyleRes 기본 스타일

순으로 처리해서 기본 스타일을 적용하고 그래도 없다면 defaultColor를 쓰는 방식으로 호환성을 보장하면서 커스텀 속성을 효율적으로 적용할 수 있다.

### 커스텀 뷰의 기본 생성자에서 @JvmOverloads를 사용할 때 왜 주의해야 하나요?
@JvmOverloads 어노테이션은 Kotlin 함수 또는 클래스에 대해 여러 오버로드된 메서드 또는 생성자를 자동으로 생성하여 Kotlin과 Java 간의 상호 운용성을 단순화하는 기능이다.    
Java는 기본값을 제공하지 ㅇ낳아 Kotlin에서 기본값을 갖는 파라미터가 있을 때 유용하다.   

```kotlin
class ElasticTextInputEditText @JvmOverloads constructor(
    context: Context, 
    attrs: AttributeSet? = null, 
    defStyle: Int = 0
) 
```
예를 들어 `defStyle`값에 기본 스타일을 적용하고자 하는 목적으로 위와 같이 선언할 경우에는 단순히 0이란 값이 하드코딩되어 의도한 스타일링을 잃을 수 있다.   
그렇기에 적절한 매개변수 기본값을 찾아 작성해주어야 한다.(ex.R.attr.editTextStyle)

# Q37 Canvas란 무엇이며 어떻게 활용하나요?
Canvas는 커스텀 드로잉을 위한 핵심 구성 요소로 화면이나 Bitmap과 같은 다른 드로잉이 가능한 표면에 직접 그래픽을 렌더링하기 위한 인터페이스를 제공한다.

| **Canvas 작동 방식**   
Canvas 클래스는 도형, 텍스트 등을 그릴 수 있는 2D 드로잉 표면을 나타낸다.   
그려진 콘텐츠가 어떻게 보여야 하는지를 정의하는 Paint 클래스와 긴밀히 상호작용한다.   
Custom View의 `onDraw()`를 재정의하면 Canvas 객체가 전달되어 무엇을 그릴지 정의할 수 있다.
```kotlin
override fun onDraw(canvas: Canvas) { 
    super.onDraw(canvas)
    canvas.drawCircle(...)
}
```

| **Canvas의 일반적인 작업**   
+ 도형(Shapes) : `drawCircle()`, `drawRect()`, `drawLine()`과 같은 메소드를 사용해 도형을 그릴 수 있다.
+ 텍스트(Text) : `drawText()` 메소드는 지정된 좌표와 모양으로 텍스트를 렌더링한다.
+ 이미지(Image) : `drawBitmap()`을 사용해 이미지를 렌더링한다.
+ 커스텀 패스(Custom Paths) : Path 객체와 `drawPath()`를 결합하여 복잡한 모양을 그릴 수 있다.

| **변환(Transformation)**   
Canvas는 크기 조절, 회전, 이동 등의 변환을 지원하여 캔버스의 좌표계를 수정해 복잡한 장면을 더 쉽게 그릴 수 있게 한다.
+ 이동(Translation) : `canvas.translate(dx, dy)`를 사용해 캔버스 원점을 새 위치로 이동한다.
+ 크기 조절(Scaling) : `canvas.scale(sx, sy)`를 사용해 드로잉 크기를 배율로 조절한다. 
+ 회전(Rotation) : `canvas.rotate(degrees)`를 사용해 지정된 각도로 캔버스를 회전한다.

이러한 변환은 메서드를 호출할 때마다 누적되어 적용되며 이후 모든 드로잉 작업에 영향을 미친다.

| **사용 사례**   
1. Custom Views : 표준적으로 제공되는 컴포넌트로 달성할 수 없는 고유한 UI 컴포넌트 만들기
2. Games : 정밀한 제어로 게임 그래픽 렌더링
3. Charts and Diagrams : 커스텀 형식으로 데이터를 시각화 해야하는 경우
4. Image Processing : 프로그래밍 방식으로 이미지 수정 또는 결합이 필요한 경우

# Q38 View 시스템의 무효화(invalidation)란 무엇인가요?
무효화(invalidation)은 View를 다시 그려야 함을 표시하는 프로세스를 의미한다.   
이는 변경 사항이 발생할 때마다 UI를 업데이트하기 위해 안드로이드 View 시스템에서 사용되는 기본 메커니즘이다.   
View가 무효화되면 시스템은 다음 드로잉 주기 동안 화면의 해당 부분을 새로 고쳐야 함을 인지하고 새롭게 그린다.

| **무효화 작동 방식**   
View에서 `invalidate()` 또는 `postInvalidate()`와 같은 메서드를 호출하면 무효화 프로세스가 트리거된다.   
시스템은 View를 dirty로 플래그 지정하며, 이는 다시 그려야 함을 의미한다.   
다음 프레임 동안 시스템은 무효화된 View를 드로잉 패스에 포함시켜 시각적 표현을 업데이트 한다.   

> 무효화(invalidation) 프로세스가 필요한 이유는 View에 변화가 필요할 때마다, 새롭게 View를 다시 랜더링 하면 성능이 크게 저하될 수 있기에 개발자/시스템에 의해 적절한 순간에 업데이트 되어야 한다.
> Jetpack Compose는 이러한 무효화 프로세스가 없고 상태에 따라 자동으로 UI가 업데이트되기에 더 주의를 해야한다.

| **무효화를 위한 주요 메소드**   
1. `invalidate()` : 단일 View 무효화하는데 사용, View를 더티로 표시하여 시스템이 다음 레이아웃 패스 중에 다시 그리도록 신호를 보낸다. View를 즉시 그리는게 아닌 다음 프레임을 위해 예약
2. `invalidate(Rect dirty)` : `invalidate()`가 오버로드된 버전으로 다시 그려야 하는 View 내의 특정 직사각형 영역 지정 가능, 더 작은 부분으로 다시 그리기를 제한하여 성능 최적화
3. `postInvalidate()` : 백그라운드 스레드에서 View를 무효화하는데 사용되고, 무효화 요청을 메인 스레드에 게시하여 스레드 안정성을 보장한다.

| **무효화 모범 사례**   
+ View의 특정 영역만 다시 그려야할 때 부분 업데이트를 위해 `invalidate(Rect dirty)`를 사용하여 변경되지 않은 영역의 불필요한 다시 그리그를 피하여 성능을 향상시킨다.
+ 애니메이션이나 복잡한 레이아웃에서 성능 병목 현상을 방지하기 위해 `invalidate()`를 자주 또는 불필요하게 호출하지 않는다.
+ 백그라운드 스레드에서의 무효화 요청에는 `postInvalidate()`를 사용하여 업데이트가 메인 스레드에서 안전하게 발생하도록 보장한다. 

## 실전 질문
Q) invalidate() 메서드는 어떻게 작동하며 postInvalidate()와 어떻게 다른가요? 각각이 적합한 실제 사용 사례를 제시해주세요.   
A) invalidate() 메소드는 View를 더티로 표시하여 다음 레이아웃 패스에서 다시 그려지도록 신호를 보낸다. invalidate()는 현재 스레드에서 즉시 뷰 무효화를 요청하고 postInvalidate는 무효화 요청을 메시지 큐에 넣고 이것을 이후 뷰 시스템이 처리한다.   
invalidate는 주로 버튼 눌렀을 때 색이 변하는 것과 같은 사용자 입력으로 즉시 피드백되어야할때 사용하고 postInvalidate는 웹에서 이미지를 다운로드하고 다 다운이 완료되면 이미지 내용을 업데이트할 때 주로 사용한다.

Q) 백그라운드 스레드에서 UI 요소를 업데이트해야 하는 경우, 다시 그리기 작업이 메인 스레드에서 안전하게 수행되도록 어떻게 보장할 수 있나요?   
A) `postInvalidate()`를 사용한다. postInvalidate는 메인 스레드의 메시지 큐에 다시 그리기 요청을 넣어 메인 스레드에서 안전하게 수행되도록 보장한다.

참고   
`invalidate(Rect dirty)`
```kotlin
val dirtyRect = Rect().apply {
    oldRect.roundOut(this) 
   this.union(newRect.roundToInt())
}

invalidate(dirtyRect)
```

# Q39 ConstraintLayout이란 무엇인가요?
ConstraintLayout은 다른 View나 부모 컨테이너에 상대적인 제약 조건(Constraints)를 사용하여 뷰의 위치와 크기를 정의할 수 있다.   
이를 통해 깊게 중첩된 뷰 계층 구조가 필요없어 성능과 코드 가독성이 향상된다.

| **ConstraintLayout의 주요 특징**   
1. 제약 조건을 이용한 위치 지정 : View는 정렬, 중앙 배치 및 앵커링을 위한 제약 조건을 사용하여 형제 View 또는 부모 레이아웃에 상대적으로 위치 지정될 수 있다.
2. 유연한 크기 제어 : `match_constraint`, `wrap_content` 및 고정 크기와 같은 옵션을 제공하여 반응형 레이아웃을 쉽게 디자인 할수 있다.
3. 체인(Chain) 및 가이드라인(Guideline) 지원 : 체인을 사용하면 뷰를 동일한 간격으로 가로 또는 세로로 그룹화할 수 있으며, 가이드라인을 사용하면 고정 또는 백분율 기반 위치에 정렬할 수 있다.
4. 배리어(Barrier) 및 그룹핑(Grouping) : 배리어는 참조된 뷰의 크기에 따라 동적으로 조정되며, 그룹핑은 여러 뷰의 가기성 변화를 단순화한다.
5. 성능 향상 : 여러 중첩 레이아웃의 필요성을 줄여 레이아웃 렌더링 속도를 높이고 앱 성능을 향상시킨다.

| **ConstraintLayout의 장점**   
1. 플랫 뷰 계층 구조 : 중첩된 LinearLayout이나 RelativeLayout과 달리 ConstraintLayout은 플랫 계층 구조를 가능하게 하여 렌더링 성능을 향상시키고 레이아웃 관리를 단순화한다.
2. 반응형 디자인 : 백분율 기반 제약 조건 및 배리어와 같은 도구를 제공하여 다양한 화면 크기 및 방향에 맞게 레아이웃을 조정한다.
3. 내장 도구 : Android Studio의 Layout Editor는 시각적 디자인 인터페이스로 ConstraintLayout을 지원하여 제약 조건을 쉽게 만들고 조정할 수 있다.
4. 고급 기능 : 체인, 가이드라인 및 배리어는 추가 코드나 중첩 레이아웃 없이 복잡한 UI 디자인을 단순화한다.

| **ConstraintLayout의 한계**   
1. 단순 레이아웃에 대한 복잡성 : LinearLayout이나 FrameLayout으로 충분할 수 있는 단순한 Layout에선 과할 수 있다.
2. 학습 곡선 : 제약 조건 및 고급 기능에 대한 이해가 필요해 학습 곡선이 가파를 수 있다.

| **ConstraintLayout 사용 사례**   
1. 반응형 UI : 다양한 화면 크기에서 정밀한 정렬과 적응성이 필요한 디자인에 이상적
2. 복잡한 레이아웃 : 여러 겹치는 요소나 복잡한 위치 지정 요구 사항이 있는 UI에 적합
3. 성능 최적화 : 중첩된 뷰 계층 구조를 단일 플랫 구조로 대체하여 레이아웃 최적화에 도움

## 실전 질문
Q) ConstraintLayout은 중첩된 LinearLayout 및 RelativeLayout과 비교하여 성능을 어떻게 향상시키나요? ConstraintLayout 사용이 더 효율적인 시나리오를 말씀해주세요.   
A) Measure와 Layout 단계에서 비용이 많이 드는데 이는 뷰 계층의 깊이에 비례한다. RelativeLayout은 중첩을 줄일 수는 있지만 Measure를 추가적으로 수행하는데 ConstraintLayout은 플랫 계층 구조를 가능하게 하여 렌더링 성능을 높인다.   
다국어 컨텐츠를 지원할때 효율적이다. 같은 의미의 말이라도 다른 언어에서는 그 단어의 길이가 바뀌기 때문에 ConstraintLayout의 배리어를 이용하면 늘 텍스트 끝에 버튼 배치를 보장할 수 있다.

Q) ConstraintLayout에서 match_constraint (0dp) 동작이 어떻게 작동하는지 설명해주세요. wrap_content 및 match_parent와 어떻게 다르며, 어떤 상황에서 사용해야 하나요?   
A) ConstraintLayout에서 0dp는 제약 기반의 동적인 크기 지정 방식을 의미하며 필수적으로 설정해야할 양쪽 끝의 제약조건 사이의 가용 공간을 모두 채운다.   
wrap_content는 뷰 내용물의 크기에 맞게 크기를 설정하여 내용물의 크기에 결정되며 match_parent는 부모 뷰를 채우기에 부모의 크기를 따른다. match_constraint (0dp)는 제약 조건 사이의 가용 공간을 모두 채우는 것에서 각각이 다르다.   
화면 양쪽 끝이나 다른 UI 요소 사이의 공간을 정확히 채워야할 때 사용한다.

참고   
배리어(Barrier) : 여러 뷰가 있을 때 그중 어떤 뷰가 크기가 변하든 항상 그 경계선 바깥쪽에 다른 뷰를 배치할 수 있도록 가변적인 마지노선을 만드는 Helper 객체

# Q40 SurfaceView 대신 TextureView는 언제 사용해야 하나요?
| **SurfaceView**
+ 별도의 스레드에서 렌더링이 처리되는 시나리오를 위해 설계된 특수한 View로 전용 드로잉 Surface를 제공한다.   
+ 주로 비디오 재생, 게임 등 성능이 중요한 작업에 쓰인다.   
+ 메인 UI 스레드 외부에 별도의 Surface를 생성해 다른 UI 작업을 차단하지 않고 효율적인 렌더링을 가능하게 한다.   
+ Surface는 SurfaceHolder 콜백 메소드로 생성/관리되고 필요에 따라 렌더링을 시작하고 중지할 수 있다.   
+ 연속적인 렌더링에 효율적이나 회전/크기 조절같은 변환에 제한이 있어 동적 상호 작용이 요구되는 UI에는 적합하지 않다.

| **TextureView**   
+ 콘텐츠를 오프스크린으로 렌더링하면서도 UI 계층 구조에 원활하게 통합된다.
+ 회전, 크기 조절, 알파 블렌딩과 같은 기능을 허용하여 변환거나 애니메이션화할 수 있다.
+ 라이브 카메라 피드를 표시하거나 커스텀 변환으로 비디오를 렌더링하는 등의 작업에서 사용
+ **메인 스레드에서 작동** -> 연속 렌더링에서는 성능적인 측면에서 덜 효율적이나 다른 UI 컴포넌트와의 상호작용과 실시간 변환에 유용하다.
+ 비디오 스트림 애니메이션, UI 내에서 콘텐츠의 시각적 변환이 필요할 때 주로 사용

| **SurfaceView와 TextureView의 차이점**   
SurfaceView는 별도의 스레드에서 작동하여 비디오 재생이나 게임과 같은 연속 렌더링 작업에 효율적이다.   
또한 렌더링을 위한 별도의 Window를 생성해 성능을 보장하지만 변환/애니메이션화 능력은 제한된다.   
TextureView는 다른 UI 컴포넌트와 동일한 Window를 공유하여 크기 조절, 회전/애니메이션이 가능하여 UI 관련 사용 사례에 더 유연하다.   
그러나 메인 스레드에서 작동하므로 고성능 렌더링이 필요한 작업에서는 효율적이지 않을 수 있다.

## 실전 질문
Q) 효율적인 리소스 관리 및 메모리 누수 방지를 위해 SurfaceView의 생명주기를 어떻게 적절하게 관리해야 하나요?   
A) SurfaceHolder 콜백을 사용해서 관리한다. Activity가 onPause가 되었을 때 렌더링 스레드에서 일시 정지를 수행한다. 또한 surfaceDestroyed 메소드에서 렌더링 스레드가 종료될 때까지 기다리고 참조를 해제함으로서 스레드 종료를 보장하고 메모리 누수를 방지한다.

Q) 카메라 미리보기를 회전 및 크기 조절과 같은 UI 변환과 함께 표시해야 하는 요구 사항이 주어졌을 때, SurfaceView와 TextureView 중 어떤 컴포넌트를 선택하는 것이 적합한가요?   
A) SurfaceView는 별도의 Window를 사용하기에 회전이나 크기 조절같은 변형에 제한이 있어 TextureView를 사용하는 것이 적합하다.
